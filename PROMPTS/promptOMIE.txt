Você é um engenheiro sênior full-stack responsável por ajustar a integração OMIE no sistema KyberRock.

Contexto do sistema:
O KyberRock é um sistema desktop Windows offline-first para operação de pedreira, com controle de entrada e saída de caminhões na balança. O OMIE continuará sendo o sistema oficial para cadastros comerciais, produtos e financeiro. O KyberRock deve manter uma cópia local sincronizada desses dados para operar mesmo sem internet.

Situação atual:

* A integração com OMIE já existe parcialmente.
* O sistema já importou aproximadamente 1187 clientes do OMIE.
* O sistema já importou produtos do OMIE.
* Ainda não está completa a sincronização de:

  * transportadoras;
  * motoristas;
  * veículos;
  * formas/condições de pagamento, se ainda não estiverem completas;
  * tabelas de preço, se aplicável;
  * contas a receber/resumo financeiro por cliente, se aplicável.
* A parte financeira continuará sendo feita no OMIE.
* O KyberRock não deve substituir o OMIE no financeiro.
* O KyberRock deve apenas consultar e espelhar dados necessários para a operação da balança.

Objetivo:
Implementar ou refatorar uma sincronização mestre de cadastros do OMIE, centralizada, segura, idempotente e offline-first.

Nome sugerido da função principal:
`syncOmieMasterData`

Ou, se o projeto estiver usando nomes em português:
`sincronizarCadastrosOmie`

Essa função deve coordenar a sincronização de todos os cadastros necessários para o KyberRock.

Escopo obrigatório:

1. Clientes

   * Reaproveitar a sincronização existente.
   * Não quebrar os 1187 clientes já importados.
   * Garantir upsert por ID externo do OMIE.
   * Atualizar dados locais quando houver alteração no OMIE.
   * Preservar campos locais que não pertencem ao OMIE.

2. Produtos

   * Reaproveitar a sincronização existente.
   * Garantir upsert por ID externo do OMIE.
   * Usar os produtos locais sincronizados nas telas operacionais.

3. Transportadoras

   * Implementar sincronização de transportadoras a partir da estrutura disponível no OMIE/projeto.
   * Verificar primeiro como o projeto atual representa clientes, fornecedores e transportadoras.
   * Não assumir hard-code sem checar a estrutura existente.
   * Criar uma regra configurável para identificar transportadoras, por exemplo:

     * tag no OMIE;
     * característica no OMIE;
     * tipo de cadastro;
     * campo específico já existente;
     * referência em cliente/pedido, se houver.
   * Salvar transportadoras em tabela/entidade local própria ou adaptar a entidade existente, conforme a arquitetura do projeto.

4. Motoristas

   * Implementar sincronização de motoristas.
   * Antes de codar, investigar no projeto e na integração atual onde motoristas podem estar vindo do OMIE.
   * Se o OMIE não tiver endpoint direto já implementado no projeto, criar uma camada configurável para origem dos motoristas.
   * Possíveis origens:

     * contatos do cliente/transportadora;
     * características;
     * tags;
     * observações estruturadas;
     * outro endpoint já usado no projeto.
   * Não inventar endpoint inexistente.
   * Se a origem exata ainda não estiver configurada, deixar a estrutura pronta e documentar claramente o ponto de configuração.

5. Veículos

   * Implementar sincronização de veículos.
   * Antes de codar, investigar como veículos estão ou deveriam estar representados no projeto.
   * Considerar que veículos podem estar no OMIE como:

     * características;
     * tags;
     * observações estruturadas;
     * dados vinculados a transportadora/cliente;
     * outro endpoint já disponível.
   * Criar mapeamento configurável para placa, tipo de veículo, transportadora e motorista padrão.
   * Não usar parsing frágil sem isolar em função própria e sem logs.
   * A placa deve ser normalizada para comparação, removendo espaços e padronizando maiúsculas.

6. Formas e condições de pagamento

   * Verificar se já existe sincronização.
   * Se existir, reaproveitar e padronizar no fluxo mestre.
   * Se não existir, implementar como subfunção da sincronização OMIE.
   * Esses dados devem ser somente leitura no KyberRock, salvo se o projeto já tiver regra local diferente.

7. Tabelas de preço

   * Verificar se o KyberRock precisa usar tabela de preço do OMIE no fluxo operacional.
   * Se já existir estrutura no projeto, integrar ao fluxo mestre.
   * Se não existir, preparar a sincronização sem quebrar a operação atual.

8. Contas a receber / situação financeira do cliente

   * O KyberRock não fará financeiro.
   * O sistema deve apenas consultar/espelhar informações necessárias para bloqueio operacional, se o PRD ou código atual já tiver essa regra.
   * Consolidar por cliente, quando aplicável:

     * total em aberto;
     * total vencido;
     * quantidade de títulos vencidos;
     * data da última atualização;
     * status de bloqueio, se existir no OMIE ou no projeto.

Arquitetura esperada:

Criar ou reaproveitar uma camada:

`OmieClient`

Responsável por:

* autenticação;
* chamadas HTTP;
* tratamento de erros;
* paginação;
* retry controlado, se já existir padrão no projeto;
* padronização de respostas.

Criar ou reaproveitar uma camada:

`OmieSyncService`

Responsável por:

* orquestrar a sincronização;
* executar subfunções por entidade;
* controlar lock de sincronização;
* registrar logs;
* salvar histórico de execução;
* fazer upsert no banco local;
* salvar payload bruto para diagnóstico.

Funções sugeridas:

```ts

  mode?: 'full' | 'incremental';
  triggeredBy?: 'manual' | 'automatic' | 'startup';

async function syncOmieProdutos(): Promise<EntitySyncResult>
async function syncOmieTransportadoras(): Promise<EntitySyncResult>

async function syncOmieVeiculos(): Promise<EntitySyncResult>

async function syncOmieCondicoesPagamento(): Promise<EntitySyncResult>
```

Regras técnicas obrigatórias:

1. Não espalhar chamadas OMIE pelas telas.
2. Telas operacionais devem usar apenas o banco local.
3. O sistema deve continuar funcionando offline com o último espelho local sincronizado.
4. A sincronização deve ser paginada.
5. A sincronização deve ser idempotente.
6. Usar upsert por ID externo do OMIE.
7. Não duplicar clientes, produtos, transportadoras, motoristas ou veículos.
8. Não apagar dados locais sem regra explícita.
9. Não sobrescrever campos locais exclusivos do KyberRock.
10. Salvar `omieId`, `omieCodigoIntegracao` ou identificador equivalente disponível.
11. Salvar data da última sincronização por registro.
12. Salvar origem do dado, por exemplo:

    * `OMIE_CLIENTE`;
    * `OMIE_PRODUTO`;
    * `OMIE_TRANSPORTADORA`;
    * `OMIE_CONTATO`;
    * `OMIE_CARACTERISTICA`;
    * `LOCAL`.
13. Salvar payload bruto do OMIE para auditoria/debug, quando possível.
14. Criar lock para impedir duas sincronizações simultâneas.
15. Tratar erro por entidade sem corromper a sincronização inteira.
16. Registrar logs técnicos úteis.
17. Não expor segredo, app_key ou app_secret em logs.
18. Não hard-code para um cliente específico.
19. Não implementar financeiro dentro do KyberRock.
20. Não criar alterações fora desse escopo.
